# Recommendations — 2026-06-26

## 待验证功能

### 1. Fast Talks 自动模式调优
- **功能**: v2026.6.10 的 #85104 — 短对话自动启用 fast mode
- **现状**: `fastAutoOnSeconds: 60` 已设置，但所有 session 的 effectiveFastMode 均为 false
- **建议**: 检查 fast mode 的触发条件。cron isolated session 的生命周期可能不符合 "短对话" 的定义（它们是单 turn 但可能运行 >60s）。如果 fastAutoOnSeconds 指的是 idle time 而非 total runtime，那对 cron job 不适用是正常的。
- **Action**: 查阅文档确认 fastAutoOnSeconds 的语义，如果是 idle-based 则标记为 N/A for cron

### 2. Cron delivery failureDestination
- **功能**: cron job 支持 `delivery.failureDestination` 配置
- **现状**: 我们的 cron job 失败时没有主动通知。dreaming-narrative-light 今天失败了，如果不是 patrol 检查到，我们不会知道。
- **建议**: 为关键 cron job（channel-patrol, cove-patrol, work-loop）配置 failureDestination 到 #kagura-dm 或一个 ops channel
- **Priority**: Medium — 目前失败率低，但随着 job 数量增多，silent failure 风险上升

### 3. Cron failureAlert
- **功能**: cron schema 支持 `failureAlert` 对象 — after N failures 触发 alert
- **现状**: 未配置
- **建议**: 设置 `failureAlert: { after: 2, mode: "announce", channel: "discord", to: "channel:<ops-channel>" }` 对重要 job。这比 failureDestination 更智能（避免偶发 1 次失败就告警）
- **Priority**: Medium

## 暂不采用

### contextMessages for cron
- cron job 支持 `contextMessages: 0-10` 将之前的 messages 作为 context
- 我们目前 cron job 都是 stateless isolated，不需要上下文。但如果将来有 "continuation" 类型的 cron（比如每天接续昨天的工作），这个功能会有用。
- **Status**: Watch — 等有合适场景再采用

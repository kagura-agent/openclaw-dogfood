# 采用建议 — 2026-07-10

## 1. Cron `failureAlert` 配置

**场景**：work-loop 的 plan-review subagent 偶发失败，目前依赖 channel-patrol 的小时巡检被动发现。

**建议**：为 work-loop cron 添加 `failureAlert` 配置：
```json
{
  "failureAlert": {
    "mode": "announce",
    "channel": "channel:1491602968741413039",
    "after": 1,
    "cooldownMs": 1800000
  }
}
```
这样 subagent 失败时立即在 kagura-dm 告警，不用等下一轮巡检。

**优先级**：中。目前 workflow 层面有重试机制，但主动告警能缩短反馈环。

## 2. `contextMessages` 用于有状态 Cron

**场景**：study-loop 的 followup 需要知道上次 scout 发现了什么。目前通过文件系统传递状态（TODO.md, targets.md）。

**建议**：如果 cron 支持 session-bound 模式（`sessionTarget: "session:<id>"`），可以让 followup 直接看到同 session 的上下文，减少文件 I/O 和 context 重建开销。

**优先级**：低。当前文件系统方案工作正常，且跨 run 的文件持久化比 session context 更可靠。

## 3. Teams Relay 稳定性监控

**场景**：Teams relay 刚上线，需要观察期。

**建议**：为 Teams relay 的关键会话添加一个轻量 health-check cron（每 4h ping 一次看 session 是否存活），类似 community-ops 的 HEARTBEAT 模式。等稳定 1 周后降频或移除。

**优先级**：中。新集成的前两周是问题高发期。

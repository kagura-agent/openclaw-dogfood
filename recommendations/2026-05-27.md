# 采用建议 — 2026-05-27

## 使用观察（过去 24h）

### ✅ 运行顺畅的功能

#### 1. Isolated Cron 大规模并发
- 今晨 9:00 AM 窗口同时触发 6 个 cron（server-health, auto-trader, cove-patrol, channel-patrol, version-check, community-ops），全部成功完成
- 24h 内 20+ 个 isolated cron session 正常运行，0 error（consecutiveErrors: 0 全线）
- **结论**: 2026.5.22 的 cron retry 增强在生产环境持续验证

#### 2. FlowForge + Cron 协作
- study-loop cron 成功驱动 FlowForge workflow，产出 wiki/scripts/regen-l1.sh 自动化脚本
- work-loop cron 驱动 github-contribution channel 的 PR review 工作
- **结论**: cron → FlowForge → subagent 三层调度链路稳定

#### 3. ACP Subagent 产出
- github-contribution channel 内 subagent 完成了 NemoClaw #4236 的 study report（耗时 ~3 min）
- 输出质量好：精确到行号的改动范围 + 风险评估
- **结论**: sessions_spawn + ACP 路径可靠

#### 4. lightContext 节省 token
- 所有 cron job 使用 `lightContext: true`，token 使用高效（大部分 job 27k-50k tokens）
- 重度 job（daily-audit 107k, morning-briefing 105k, github-contribution 124k）合理偏高

### ⚠️ 摩擦点

#### 1. Email Patrol 信息密度高
- email-patrol 输出 23 封邮件摘要，格式良好但长度接近 Discord 2000 字符限制
- 考虑加 `presentation` blocks 或分条消息

#### 2. 无跨 Cron Session 上下文共享
- daily-audit 提到 gogetajob broken 4 天，但 work-loop / github-check 独立判断
- 目前靠 memory/ wiki 间接共享，无 session 间直接引用机制
- 这是已知的架构限制，不是 bug

### 💡 建议探索

#### 1. Cron failureAlert 配置（中优先级）
- 当前所有 cron 无 failureAlert 配置
- 建议为关键 job（server-health, auto-trader, email-patrol）配置 `failureAlert.after: 2`
- 效果：连续 2 次失败时自动通知到 #evolution channel

#### 2. Session Key 复用（低优先级）
- 部分 cron（如 daily-audit, morning-briefing）每次都是全新 isolated session
- 考虑用 `sessionTarget: "session:daily-audit"` 让它们保留上次上下文
- 风险：上下文膨胀；需要配合 `lightContext`

#### 3. Delivery threadId（低优先级）
- 高频输出的 cron（email-patrol, auto-trader）可以用 threadId 投递到固定 thread
- 减少 channel 噪音，保持主 channel 干净

## 不采用

### Webhook Delivery
- 理由：当前所有输出都在 Discord 内消费，无外部系统需要接收

### wakeMode: next-heartbeat
- 理由：所有 cron 用 `wakeMode: now` 即时执行，无需等 heartbeat
